  <action>
1. Create `supabase/schema.sql` with the complete database schema from RESEARCH.md:

   **Tables:**
   - `alberta_config`: id (UUID PK), effective_from (DATE NOT NULL), effective_to (DATE nullable — NULL means currently active), grid_emission_factor (NUMERIC), payout_percentage_low (NUMERIC), payout_percentage_high (NUMERIC), cash_back_rate (NUMERIC), default_escalation_rate (NUMERIC), default_net_metering_buy_rate (NUMERIC), default_net_metering_sell_rate (NUMERIC), created_at (TIMESTAMPTZ), notes (TEXT)
   - `carbon_benchmark_schedule`: id (UUID PK), config_id (UUID FK → alberta_config ON DELETE CASCADE), year (INTEGER), price_per_tonne (NUMERIC), UNIQUE(config_id, year)
   - `proposals`: id (UUID PK), created_at (TIMESTAMPTZ), updated_at (TIMESTAMPTZ), config_id (UUID FK → alberta_config), inputs (JSONB NOT NULL), outputs (JSONB nullable), status (TEXT DEFAULT 'draft')

   **RLS Policies:**
   - Enable RLS on all three tables
   - alberta_config: public SELECT (no auth needed in v1)
   - carbon_benchmark_schedule: public SELECT
   - proposals: public SELECT, INSERT, UPDATE (no auth in v1)

   **Indexes:**
   - `idx_alberta_config_effective` on alberta_config(effective_from DESC)

2. Create `supabase/seed.sql` with initial data:

   **Alberta config seed:**
   - effective_from: '2025-01-01', effective_to: NULL
   - grid_emission_factor: 0.55 (t CO2/MWh)
   - payout_percentage_low: 0.45 (45%)
   - payout_percentage_high: 0.65 (65%)
   - cash_back_rate: 0.03 (3%)
   - default_escalation_rate: 0.05 (5%)
   - default_net_metering_buy_rate: 0.168 ($/kWh)
   - default_net_metering_sell_rate: 0.335 ($/kWh)
   - notes: 'Initial Alberta config based on calculations document'

   **Carbon benchmark schedule seed** (linked to the inserted config_id):
   - 2025: $80/tonne
   - 2026: $95/tonne
   - 2027: $110/tonne
   - 2028: $125/tonne
   - 2029: $140/tonne
   - 2030: $170/tonne

   Use a CTE or subquery to reference the config_id from the INSERT.

3. Create `src/lib/supabase/server.ts`:
   - Use `@supabase/ssr` createServerClient pattern for Next.js 16
   - MUST `await cookies()` (async in Next.js 16 — breaking change)
   - Export async `createClient()` function
   - Read from `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` env vars

4. Create `src/lib/supabase/client.ts`:
   - Use `@supabase/ssr` createBrowserClient
   - Export `createClient()` function
   - Read from same env vars

5. Create `src/lib/supabase/types.ts`:
   - Placeholder for Supabase generated types
   - Export a `Database` type stub that will be replaced by `supabase gen types typescript` after schema is applied
   - Include a comment explaining how to regenerate: `npx supabase gen types typescript --project-id YOUR_PROJECT_ID > src/lib/supabase/types.ts`

6. Create `src/lib/config/alberta.ts`:
   - Export `getAlbertaConfig(asOfDate?: Date): Promise<AlbertaConfig>`
   - Uses server Supabase client to query `alberta_config` table
   - Filters by effective date: `.lte('effective_from', dateStr).or('effective_to.is.null,effective_to.gte.' + dateStr)`
   - Orders by `effective_from DESC`, limits to 1 (most recent applicable config)
   - Fetches related `carbon_benchmark_schedule` rows for the config
   - Converts NUMERIC database values to Decimal instances using `d()` helper
   - Returns typed `AlbertaConfig` object

   **Critical pitfall:** Supabase returns NUMERIC as strings in JS. This is actually GOOD — pass directly to `d()` constructor. Never parse through float first.

7. Update `.env.local.example` to ensure it has both Supabase vars clearly documented with instructions.
